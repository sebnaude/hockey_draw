# spec-032 — Constraint group restructure: `symmetry_breakers` (always-on) + `core` minus spacing + lonesome `spacing`

**Spec source:** convenor request (this session) — "create some new groupings with flags. An easy grouping is `--symmetry-breakers` … always applied by default … the two NIHCFill ones and the lexicographical ordering. … redo `--core`, it should contain all that's currently in core except drop the equalspacingconstraint. That should leave three distinct groups — core, symmetry breakers, soft — and the lonesome equalspacingconstraint." Confirmed decisions this session: spacing stays applied in a plain/simple solve; symmetry-breakers are always-on but with a `--no-symmetry-breakers` escape.

## Why

spec-023 gave us composable `--groups`, but every production constraint currently sits in just two primary buckets (`core`, `soft`), with the symmetry-breakers and matchup-spacing folded into them. The convenor wants three cleaner, independently-selectable concerns plus one lonesome constraint:

- The **symmetry breakers** (`NIHCFillWFBeforeEF`, `NIHCFillEFBeforeSF`, `SoftLexMatchupOrdering`) are pure objective-shaping tie-breakers — they should apply in *every* solve regardless of which `--groups` are chosen, not be bundled inside `soft` where `--groups core` would drop them. They are currently tagged `{soft, soft_optimisation}` (`registry.py:311-332,447-454`).
- **`EqualMatchUpSpacing`** is the one rule the convenor wants able to peel off `core` independently (it is the most slack-sensitive, often-relaxed rule). It is currently `{core, critical_feasibility}` (`registry.py:168-176`) and already has a `regen_soft` sibling (`EqualMatchUpSpacingRegenSoft`), so its regen story is unaffected.

Net target group structure for the *applied solver* constraints: **`core`** (everything currently core except spacing), **`symmetry_breakers`** (always-on, 3 atoms), **`soft`** (the rest of the optimisation softs), and **`spacing`** (the lone `EqualMatchUpSpacing`).

Crucially (confirmed this session): a **plain solve applies everything** because the no-`--groups` path runs `DEFAULT_STAGES` directly (`run.py:860` sets `staged_constraint_filter = None`), *not* group resolution. So retagging only changes what an explicit `--groups core` selects — it does **not** remove spacing or symmetry-breakers from a normal solve.

(review fix — C1: The original plan claimed "Giving spacing its own non-empty `{spacing}` tag keeps it inside the derived `default`/`all`/`production` groups (membership = 'has any group')." This is WRONG. The real predicate in `registry.py:879` is `_is_fresh_build = bool(info.groups & {'core', 'soft'})` — NOT `bool(info.groups)`. After retagging to `{spacing}` and `{symmetry_breakers}`, all four atoms would DROP OUT of `default`/`all`/`production`. The fix is: the `_is_fresh_build` predicate in `registry.py:879` must be widened to `bool(info.groups & {'core', 'soft', 'spacing', 'symmetry_breakers'})` so `default` membership is genuinely unchanged after the retag. This widening is Unit A work, in `registry.py`. The "Out of scope" entry about `spacing` staying in `default` is still correct in intent — the mechanism is predicate widening, not free-ride on `bool(info.groups)`.)

Cost of not doing this: `--groups core` silently includes spacing and excludes symmetry-breakers, the opposite of what the convenor wants; and there is no way to ask for "core rules without spacing" or "everything but the tie-breakers".
